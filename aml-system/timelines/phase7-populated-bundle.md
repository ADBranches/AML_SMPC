## `tests/README.md`

```md
# Phase 7 Test Suite

This directory contains the complete **Phase 7 — Testing, Benchmarking & Validation** scaffold for the AML system.

## Goals

Phase 7 confirms that the system is:

- technically correct,
- operationally measurable,
- compliance-aligned,
- and ready to produce evidence for the report and final demo.

## Structure

```text
tests/
├── compliance/         # Compliance proof tests, validation scripts, assertions, report templates
├── evidence/           # Captured evidence artifacts generated by test execution
├── fixtures/           # Canonical request payloads and expected outputs
├── integration/        # Functional + integration + end-to-end test scripts
├── logs/               # Raw execution logs
├── performance/        # Locust-based load testing and performance criteria
├── README.md           # This file
└── TEST_ORDER.md       # Official execution order
```

## Conventions

### 1) Separation of concerns

- **Test scripts** live under `tests/integration/`, `tests/compliance/`, and `tests/performance/`.
- **Fixtures** live under `tests/fixtures/`.
- **Logs** live under `tests/logs/`.
- **Evidence artifacts** (saved responses, snapshots, query outputs) live under `tests/evidence/`.

> Do **not** place executable scripts inside `tests/evidence/`.

### 2) Naming

Use names that reveal:

- component,
- scenario,
- and whether it is positive-path or negative-path.

Examples:

- `he_encrypt_decrypt_positive_test.sh`
- `he_invalid_input_negative_test.sh`
- `smpc_match_positive_test.sh`
- `zk_invalid_proof_test.sh`

### 3) Pass/fail rule

Every test file must define:

- feature under test,
- input fixture used,
- expected output,
- pass/fail rule,
- evidence artifact retained.

### 4) Standard environment variables

The scripts use these environment variables if present:

- `DATABASE_URL`
- `HE_GATEWAY_BASE_URL`
- `SMPC_BASE_URL`
- `ENCRYPTION_SERVICE_BASE_URL`
- `ZK_PROVER_BASE_URL`
- `REGULATOR_API_BASE_URL`

If a variable is not set, a safe local default is used inside the script.

### 5) Evidence discipline

Every executable Phase 7 test should:

- create a timestamped or named log file under `tests/logs/...`,
- save useful responses under `tests/evidence/...`,
- exit with status `0` on success,
- exit non-zero on failure.

## Recommended cleanup before running

1. Move any executable `.sh` files out of `tests/evidence/` into `tests/integration/` or `tests/compliance/`.
2. Standardize `DATABASE_URL` across your shell, CI, and demo scripts.
3. Ensure PostgreSQL is running and migrations have been applied.
4. Ensure HE, SMPC, encryption-service, zk-prover, and regulator-api are running before E2E tests.
5. Run `api_end_to_end_test.sh` before standalone zk/compliance tests if you are not using the CI wrappers, because proof generation depends on an existing transaction + audit trail.

## Phase 7 coverage

- Functional tests
- Integration / end-to-end tests
- Compliance validation tests
- Performance tests
- Evidence collation for report/demo

```

## `tests/TEST_ORDER.md`

```md
# Phase 7 Test Execution Order

To keep results organized and reproducible, Phase 7 should run in this order:

1. **Functional tests**
2. **Integration and end-to-end tests**
3. **Compliance validation tests**
4. **Performance tests**
5. **Evidence collation for report/demo**

## Why this order matters

### 1. Functional tests first

Functional tests confirm basic correctness of individual components before service chaining begins.

### 2. Integration and E2E tests second

Once individual components are stable, end-to-end service interaction can be validated with confidence.

### 3. Compliance validation third

Compliance assertions should only be validated against a system that is already behaving correctly.

### 4. Performance tests fourth

Performance measurements are only meaningful after correctness is stable.

### 5. Evidence collation last

The final step is to organize:

- logs,
- screenshots,
- endpoint responses,
- database snapshots,
- benchmark summaries,
- and compliance results

for the report, defense, and demo.

## Execution map

```text
1. HE functional tests
2. SMPC functional tests
3. zk proof generation / verification tests
4. API end-to-end tests
5. Compliance proof tests
6. Compliance validation scripts
7. Performance load tests
8. Phase 7 evidence collation
```

```

## FATF Recommendation 10

Pass if:

- a `FATF_REC10` proof exists for the transaction under test,
- the proof is retrievable,

Pass if:

- audit rows are retrievable for the transaction under test,
- a `FATF_REC11` proof exists,

Pass if:

- originator and beneficiary institution metadata are present for the selected transaction,
- a `FATF_REC16` proof exists,
- and the proof can be retrieved and verified.

## Evidence required

Each compliance validation run should retain:

- proof IDs,
- transaction IDs,
- audit retrieval results,
- verification responses,
- and a final pass/fail summary.

## `tests/compliance/rec10_proof_test.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd)"
TEST_DIR="$ROOT_DIR/tests"
TIMESTAMP="$(date -u +%Y%m%dT%H%M%SZ)"
mkdir -p "$TEST_DIR/logs" "$TEST_DIR/evidence"

DATABASE_URL="${DATABASE_URL:-postgresql://aml_dev:securepassword@localhost:5433/aml_dev}"
ZK_PROVER_BASE_URL="${ZK_PROVER_BASE_URL:-http://127.0.0.1:8084}"
LOG_DIR="$TEST_DIR/logs/compliance"
EVIDENCE_DIR="$TEST_DIR/evidence/compliance"
LOG_FILE="$LOG_DIR/rec10_proof_test_${TIMESTAMP}.log"
EVIDENCE_FILE="$EVIDENCE_DIR/rec10_proof_test_${TIMESTAMP}.json"
mkdir -p "$LOG_DIR" "$EVIDENCE_DIR"

python3 - <<'PY' "$ZK_PROVER_BASE_URL" "$LOG_FILE" "$EVIDENCE_FILE"
import json, sys, urllib.request
base_url, log_file, evidence_file = sys.argv[1:4]
req = urllib.request.Request(base_url+'/proofs/generate', data=json.dumps({'tx_id':'TX-E2E-001'}).encode(), headers={'Content-Type':'application/json'}, method='POST')
with urllib.request.urlopen(req) as resp:
    proofs = json.loads(resp.read().decode('utf-8'))
matching = [p for p in proofs if p['rule_id'] == 'FATF_REC10']
passed = len(matching) == 1 and matching[0]['verification_status'] in ('generated', 'verified')
record = {'rule_id': 'FATF_REC10', 'match_count': len(matching), 'proof': matching[0] if matching else None, 'passed': passed}
open(log_file,'w').write(json.dumps(record, indent=2))
open(evidence_file,'w').write(json.dumps(record, indent=2))
if not passed:
    raise SystemExit(1)
PY

echo "[PASS] rec10 proof test completed. Evidence: $EVIDENCE_FILE"

```

import json
from pathlib import Path
from datetime import datetime, timezone

FIXTURE = Path(__file__).resolve().parents[1] / 'fixtures' / 'performance_transactions.json'

with open(FIXTURE, 'r', encoding='utf-8') as f:

_counter = 0

def submit_transaction_payload():
    global _counter
    _counter += 1
    payload = copy.deepcopy(DATA['transactions'][0])
    payload['tx_id'] = f"{payload['tx_id']}-{_counter:06d}"
    payload['timestamp'] = datetime.now(timezone.utc).isoformat().replace('+00:00', 'Z')
    return payload
